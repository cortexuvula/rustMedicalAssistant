# v0.10.11 — Split `SettingsContent.svelte` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Decompose the 1,717-LOC `SettingsContent.svelte` monolith into a thin parent (~80 LOC) and four focused per-section components in a new `src/lib/components/settings/` subdirectory. Behavior is preserved exactly — this is a maintenance refactor, invisible to users.

**Architecture:** Five tasks, one section per commit. Order is smallest-first to prove the pattern: Prompts (~150 LOC) → General (~250 LOC + dialogs) → Models (~370 LOC) → Audio (~600 LOC + download-progress listener). Final task bumps the version. Each commit leaves the parent functional, so any regression is bisectable to a single section.

**Tech Stack:** Svelte 5 (runes, `<svelte:window>`, `onMount` / `onDestroy`), TypeScript, vitest.

**Spec:** `docs/superpowers/specs/2026-05-01-split-settings-content-design.md`

## Shared extraction recipe (applies to Tasks 1–4)

Each extraction task follows the same six-step pattern. The plan body for each task lists which **specific** state vars, handlers, markup line range, and dialogs to move; the recipe below describes **how**.

### Recipe

1. **Read the parent file** (`src/lib/components/SettingsContent.svelte`) and identify the current line range of the section's `{#if activeSection === '...'}` markup branch and the section's state declarations + handler functions in the `<script>` block.

2. **Create the new section component** at `src/lib/components/settings/<Name>.svelte` with this skeleton:

```svelte
<script lang="ts">
  // Imports (only what this section actually uses):
  import { onMount } from 'svelte';   // only if the section needs lifecycle
  // ...other imports as needed (stores, API wrappers, dialog components)

  // State declarations (moved from parent, verbatim).
  // Handler functions (moved from parent, verbatim).

  // onMount / onDestroy if any.
</script>

<!-- Markup moved from parent's {#if} branch, verbatim except for any wrapping -->
<!-- <section class="settings-section"> stays — it's the section's own root -->

<style>
  /* Scoped CSS rules moved from parent that style only this section's classes */
</style>
```

3. **Move state, handlers, and markup** from the parent into the new file. Verbatim. Do NOT change behavior, naming, or logic. Do NOT add new types, props, or refactorings.

4. **Move scoped CSS rules** that style ONLY this section's elements. For each rule in the parent's `<style>` block, ask: "Does this selector match a class used only inside this section?" If yes, move. If shared (e.g., `.form-group`, `.section-title`, `.form-label`, `.btn-primary`), leave in parent. If genuinely unclear, leave in parent and add a `/* TODO: classify */` comment so the reviewer can decide.

5. **Update the parent** (`SettingsContent.svelte`):
   - Add the import: `import <Name> from './settings/<Name>.svelte';`
   - Replace the inline `{#if activeSection === '...'} ... {/if}` markup with `<<Name> />` (the section component itself becomes the entire branch body).
   - Remove the moved state declarations, handler functions, imports that are no longer used, and CSS rules.
   - Where a section moved a dialog (e.g., `<VocabularyDialog>` from General), also remove the dialog render and its import from the parent.

6. **Verify and commit**:
```

cd /Users/cortexuvula/Development/rustMedicalAssistant/.worktrees/v0.10.11-split-settings && npx vitest run 2>&1 | tail -5
```

### Discipline reminders applying to all extraction tasks

- **No behavior changes.** This is a refactor. Don't rename a variable, don't tweak a string, don't "fix" anything you notice.
- **Each commit must leave the parent functional.** Verify by running `npm run check` and `npx vitest run` before committing.
- **CSS judgment calls go to the parent.** When in doubt, leave it in `SettingsContent.svelte`. The reviewer can move it later if needed.
- **Don't introduce props.** Children read global stores directly, matching the existing pattern.
- **Don't introduce events / callbacks.** Each section is self-contained.
- **Don't run the dev server.** The implementer verifies via `npm run check` and `npx vitest run` only. The user does the manual smoke test post-merge.
